# Orca Fork Skill

Create a new `orca-fork` skill that branches the current AI coding agent conversation and opens the branch in a new cmux tab.

## Domain Decisions

* A **conversation** is the provider-owned transcript/history. Codex calls this a thread; Claude Code calls this a session.
* A **fork** branches an existing conversation into a new conversation and opens it in its own tab.
* An **agent instance** is a live running agent type in an Orca-managed surface.
* A **worker** is a task-bearing agent instance. An open-only fork is an agent instance, but not a worker until it is given a task/prompt.

## Skill Shape

* Add `skills/orca-fork/SKILL.md`.
* The skill is separate from `orca-agent` because fork preserves conversation history while spawn starts from a fresh brief.
* The skill is a thin conversational wrapper around bundled scripts, matching the style of `orca-agent`.
* The skill must be self-contained. It must not source, execute, or reference scripts in sibling skills.

## Runtime Files

The first implementation should include:

* `skills/orca-fork/scripts/orca-fork.sh`
* `skills/orca-fork/scripts/orca-cmux.sh`
* `skills/orca-fork/scripts/orca-fork-adapter.sh` or equivalent adapter config

`orca-cmux.sh` should be an exact duplicate of the existing cmux seam when it is intended to stay shared. Tests should fail if shared helper copies drift.

## Source Conversation Selection

The source conversation id is provider-specific and cannot be used with another agent type. Do not expose a generic `--agent` plus `--conversation-id` interface.

Supported sources:

* Auto Codex: `CODEX_THREAD_ID`
* Auto Claude: a verified exact Claude session id environment variable, when available
* Explicit Codex: `--codex-thread-id <uuid>`
* Explicit Claude: `--claude-session-id <uuid>`

Rules:

* If exactly one auto source is available and no explicit source is passed, use it.
* If an explicit provider-specific source is passed, use it and ignore auto sources.
* If multiple explicit sources are passed, fail.
* If multiple auto sources are available and no explicit source is passed, fail as ambiguous.
* If no exact source is available, fail with an actionable message.
* Do not use recency fallbacks such as `codex fork --last`, `claude --continue --fork-session`, or scanning provider session files.

## Launch Commands

Codex:

```bash
codex fork <thread-id> [prompt]
```

Claude Code:

```bash
claude --resume <session-id> --fork-session [prompt]
```

Pass the optional prompt as part of the launch command rather than typing it after readiness, unless a provider CLI is later proven not to support prompt-at-launch.

## cmux Behavior

* Create a plain terminal surface in the calling workspace.
* Target surfaces by UUID only.
* Rename the tab after creation.
* If Codex asks to trust the calling workspace before showing the composer, select the "Yes" option so the fork can finish launching in the same workspace the human already invoked Orca from.
* Do not close the tab after launch has started, even on post-launch failure.
* Use fire-and-confirm behavior: launch, wait for readiness, then report.
* Do not monitor the fork after confirmation.
* Do not persist a worker registry.

## Tab Naming

* Open-only fork default: `fork-codex` or `fork-claude`
* Prompted fork: derive a short slug from the prompt or provided title, such as `fork-investigate-login`
* Support `--title <title>` override
* Return the requested tab name in `tab=<name>`
* Do not put conversation UUIDs in the tab title

## Output

On success, print `key=value` lines:

```text
status=ok
provider=codex|claude
source_conversation=<provider-specific-id>
surface=<surface-uuid>
tab=<tab-name>
prompt_sent=true|false
```

On failure, print `status=error`, include any values already known, include `error=<reason>`, and leave any created tab open.

## Tests

Add tests in the existing shell-test style:

* Package test includes `orca-fork`.
* Shared helper drift test compares intentional duplicate helpers, starting with `orca-cmux.sh`.
* Adapter tests cover provider-specific source selection and launch command construction.
* Fork integration tests use a fake cmux and fake screens to verify:
  * Codex source via `CODEX_THREAD_ID`
  * Claude explicit source via `--claude-session-id`
  * ambiguity failures
  * no recency fallback
  * Codex trust prompt handling
  * prompt-at-launch command construction
  * open-only fork readiness
  * tab rename
  * failed readiness leaves tab open and reports surface UUID
