# orca-msg skill

Create an `orca-msg` skill that delivers a follow-up message to an existing agent instance in a cmux terminal surface.

The skill is a conversational wrapper over a deterministic bundled script. The wrapper may interpret the human's instruction, gather current context, ask clarifying questions, and assemble the agent-facing message. The script does not invent content; it resolves the target, verifies readiness, sends exact text, and presses Enter.

## User interaction

The human can target an agent instance in two ways.

* Paste a cmux "copy ids" block. `orca-msg` extracts `surface_id` and ignores positional refs such as `surface_ref`, `pane_ref`, and `workspace_ref` for targeting.
* Describe a target surface, for example "the Claude agent in the aiml-services workspace." The skill searches cmux metadata in the caller's current cmux window. If the description resolves to zero or multiple plausible surfaces, the skill asks the human a follow-up question and shows candidates instead of guessing.

`orca-msg` treats the human's text as an instruction to the orchestrator, not necessarily the literal message body. For example, "give the Claude agent in aiml-services workspace all the details of this issue and ask it to give you feedback on your decision" means the skill should identify the target, gather the issue context, compose the message, and send it when unambiguous.

By default, the skill sends immediately once both the target and message content are clear. It asks before sending when the target or content is materially ambiguous.

## Target resolution

The only durable target is the surface UUID. cmux positional refs can appear in copied input and candidate displays, but must not be used for deferred targeting.

Descriptor resolution is intentionally conservative.

* Search the caller's current cmux window by default.
* Use workspace metadata such as title, custom title, and current directory to match workspace phrases.
* Use surface metadata such as terminal kind, cwd, shell, selected state, and display labels when available.
* Use descriptor words like "Claude" or "Codex" as agent-type hints.
* If the descriptor does not state an agent type, inspect recent screen text and infer Claude from `← for agents`, or Codex from the `›` input marker plus a footer containing ` · `.
* If agent type remains unknown, ask the human before sending.

An explicit pasted `surface_id` is authoritative and may bypass descriptor search.

## Readiness and blocked states

`orca-msg` only sends when the target surface appears to be at an agent input prompt.

* Claude readiness uses the adapter marker `← for agents`.
* Codex readiness uses the adapter marker `›` and the footer separator ` · `.
* The script must not answer permission prompts, trust prompts, shell prompts, merge editors, running-command states, or unknown blocked UI. In these cases it reports that the target is not ready and asks the human to bring it to a ready agent prompt.

This differs from `orca-spawn` spawn, which may answer known launch-time trust prompts because it owns that launch sequence.

## Message delivery

Short messages are typed directly into the target surface and submitted with Enter.

For long or context-heavy messages, the wrapper writes the assembled message to a temporary file outside the repository, such as `$TMPDIR/orca-msg.<random>/message.md`, leaves that file in place, and sends a short instruction containing the absolute path. The skill reports the exact sent text and the temp file path.

The delivery script should support both direct `--message` and `--message-file` inputs so the wrapper can choose the safer transport.

## Packaging

`orca-msg` is a new self-contained skill under `skills/orca-msg/`.

Expected files:

* `skills/orca-msg/SKILL.md`
* `skills/orca-msg/scripts/orca-msg.sh`
* `skills/orca-msg/scripts/orca-cmux.sh`
* Agent readiness helper or adapter data bundled inside the skill

The skill must not reference sibling skill directories or repo-root runtime scripts. Small shared helpers should be duplicated into the skill package. Package tests should include `orca-msg`, validate executable scripts, and assert copied cmux helpers stay byte-for-byte identical where they are intentionally shared.

Update local development symlinks and plugin docs so the skill appears with `orca-spawn` and `orca-fork`.

## Validation

Automated tests should use a fake cmux binary to verify:

* copy-ids input extracts `surface_id` and refuses positional refs for targeting
* descriptor resolution sends only after exactly one candidate is selected
* ambiguous descriptor output includes candidate surfaces and exits without sending
* Claude and Codex readiness markers allow sending
* blocked or unknown screen states prevent sending
* direct messages preserve text as one argument through the cmux seam
* long-message file-pointer flow sends an absolute path and leaves the file available

Manual smoke test:

* Open or identify an existing Claude or Codex agent in cmux.
* Invoke `orca-msg` with a pasted copy-ids block and a short message.
* Invoke `orca-msg` with a natural-language descriptor for the same surface.
* Confirm the agent receives the intended message and `orca-msg` reports the exact text it sent.
